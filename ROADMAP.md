# 改进方向评估与路线图

> **⚠️ 声明：以下所有特性均为远期规划，目前尚未实现。**
> 本文档用于记录对未来改进方向的评估和思考，不代表开发承诺或时间表。

> 本文档面向 RustLuaMud 的核心定位：
> **"基于 Rust + LuaJIT 的终端 MUD 客户端，面向无 GUI 环境下 7x24 小时挂机，兼容 MUSHclient 脚本 API。"**

各方向按 **收益 : 代价** 排序，评估维度说明：

| 维度 | 含义 |
|------|------|
| **收益** | 对定位的贡献度：是否真正解决"无 GUI 7x24 挂机"场景的痛点 |
| **风险** | 破坏现有功能 / 引入新 bug / 增加维护负担的可能 |
| **难度** | 预估实现工作量（含测试、文档） |
| **急迫性** | 当前版本是否值得优先做 |

评分标准：★ 最低，★★★★★ 最高

---

## 1. 守护进程模式（Daemon mode）

| 收益 | 风险 | 难度 | 急迫性 |
|:----:|:----:|:----:|:------:|
| ★★★★★ | ★ | ★★ | ★★★★★ |

**评估**：当前的最大短板。客户端无法在 ssh 断开后继续运行，这是实现"7x24 挂机"最直接的障碍。不解决这个，其他所有改进都建立在"你得一直开着终端"的前提上。

**收益**：让客户端真正 headless，配合 `systemd` 或 `screen` 托管可做到开机自启、崩溃自愈、ssh 断开不中断。

**风险**：极低。`fork() + setsid()` 是成熟的 POSIX 操作，仅影响启动路径，不触及核心逻辑。需要处理的主要是 stdout/stderr 重定向和信号处理（SIGTERM 优雅退出）。

**实现路径**：
1. `--daemon` 参数触发 fork/setsid
2. PID 文件写入 `/var/run/rustluamud.pid`
3. `--daemon stop` / `--daemon status` 通过 PID 文件管理
4. 可选：`--daemon` 模式下屏蔽终端渲染，仅写日志

**前置依赖**：无。

---

## 2. 远程控制接口（HTTP / Unix Socket API）

| 收益 | 风险 | 难度 | 急迫性 |
|:----:|:----:|:----:|:------:|
| ★★★★★ | ★★ | ★★★★ | ★★★★★ |

**评估**：让客户端可编程控制的关键接口。守护进程解决了"活着"的问题，API 解决了"怎么管"的问题。二者结合才构成完整的 headless 方案。

**收益**：
- 外部程序（web 面板、bot、shell 脚本）可查询状态、执行 Lua 代码、加载角色
- 与告警系统组合可实现自动化运维
- Lua 的 `json_encode/decode` 已就位，数据交换层现成

**风险**：中等。HTTP 服务器带来新的攻击面，需注意：
- 只绑定 `127.0.0.1`，不暴露到外网
- API token 认证（可选）
- Lua 执行接口 `/api/lua/exec` 是潜在的 RCE 入口，必须加认证

**实现路径**：
1. 引入轻量 HTTP 库（`tiny_http` 或 `axum` 或直接用 tokio 的 `TcpListener` + 手写 http 解析）
2. 启动时在 `127.0.0.1:指定端口` 监听
3. 定义 RESTful 接口（见下）
4. 通过 `mpsc` channel 与主循环通信（类似现有的 `connect_tx` / `reconnect_tx`）

**推荐 API 设计**：

```
GET  /api/status                    → 所有连接状态 JSON
GET  /api/status/<session_id>       → 单个连接状态
POST /api/switch/<session_id>       → 切换前台
POST /api/lua/exec                  → 执行 Lua 代码
  body: {"code": "return GetInfo(1)", "session": 2}
POST /api/profile/load              → 加载角色配置
  body: {"name": "mychar"}
GET  /api/log/<session_name>        → 获取最近日志行
```

**前置依赖**：守护进程模式（没有 daemon 模式时 API 还有用，但价值减半）。

## 3. 告警通知（Alerting）

| 收益 | 风险 | 难度 | 急迫性 |
|:----:|:----:|:----:|:------:|
| ★★★★ | ★ | ★★★ | ★★★★ |

**评估**：7x24 运行的必需品——出问题了你需要知道。当前断线了只会默默重连，重连失败也只在终端打印一行，没人看得见。

**收益**：
- 断线、重连失败、Lua 执行错误等事件主动推送
- 与健康监控组合可实现"无人值守"
- 推送渠道灵活（Telegram / 钉钉 / 邮件 / Webhook）

**风险**：低。告警是纯外发的异步操作，不影响核心逻辑。主要风险是网络不通导致告警发不出去——这不影响客户端本身。

**实现路径**：
1. 定义告警事件类型（断线、重连失败 N 次、Lua 错误、自定义）
2. 实现推送器 trait，支持多个后端
3. 内置 Telegram Bot 推送最简单（HTTP POST 到 `api.telegram.org`）
4. Lua API 暴露 `send_alert(msg)`，脚本可主动触发告警

**前置依赖**：守护进程模式（没有 daemon 模式时你在终端前，不需要告警）。

## 4. 连接健康监控（Health Check）

| 收益 | 风险 | 难度 | 急迫性 |
|:----:|:----:|:----:|:------:|
| ★★★★ | ★ | ★★ | ★★★ |

**评估**：当前的重连策略是"断线→等 N 秒→重连"，不判断是服务器挂了还是网络抖动，也没有被动检测（服务器没响应但 TCP 没断的情况很多 MUD 都有）。

**收益**：
- 心跳检测：N 分钟无数据时自动 ping，避免"假连接"
- 指数退避：避免被服务器 ban
- 连接质量指标暴露给 Lua，脚本可据此做决策（如"延迟太高就换线路"）

**风险**：低。心跳检测是新增逻辑，不影响现有重连流程。注意点：
- 心跳频率不能太快（建议 2-5 分钟），避免被判定为 flood
- 指数退避需要有上限（如最长 30 分钟）

**实现路径**：
1. Session 层增加"最后收到数据时间"跟踪
2. 定时器检查超时 → 发送心跳（可配置，默认 `""` 即不启用）
3. 重连逻辑改为指数退避：1s → 2s → 4s → 8s → ... → 上限
4. Lua API 暴露 `GetSessionOption("health")` 返回统计

**前置依赖**：无。

## 5. 凭据安全管理（Credentials Management）

| 收益 | 风险 | 难度 | 急迫性 |
|:----:|:----:|:----:|:------:|
| ★★★ | ★ | ★★ | ★★ |

**评估**：当前 password 明文写在 TOML 里。单人使用问题不大，多人共用服务器或打算开源分享配置模板时有安全风险。

**收益**：
- 配置文件中不出现明文密码
- 可与 `.env` 文件或系统密钥环集成
- 减少误提交密码到 git 的风险

**风险**：低。纯配置层变更，不触及核心逻辑。注意：
- 环境变量方式最简单，Rust 侧用 `std::env::var` 读取
- 不需要引入复杂的密钥管理系统

**实现路径**：
1. `config.rs` 中解析 TOML 后检查 `${VAR_NAME}` 模式
2. 匹配则替换为环境变量值
3. 记录哪些值来自环境变量（调试时不泄露值本身）

```
# 配置示例
password = "${MUD_MYCHAR_PASSWORD}"

# 启动前
export MUD_MYCHAR_PASSWORD="secret"
```

**前置依赖**：无。

## 6. 脚本热更新（Hot Reload — 精细化重载）

| 收益 | 风险 | 难度 | 急迫性 |
|:----:|:----:|:----:|:------:|
| ★★★ | ★★★ | ★★★★★ | ★★ |

**评估**：当前的 `/load reload` 是"全部重来"，对于大脚本库（如 `class/` 下的复杂脚本），重载可能丢失动态状态。精细化重载很实用，但实现复杂度和风险都不低。

**收益**：
- 修复 bug 不需要完全重载，减少对挂机状态的影响
- 只重载特定模块的 trigger/alias/timer，不影响其他

**风险**：高。精细化重载容易导致：
- 内存泄漏（旧闭包未被 GC）
- 状态不一致（部分重载导致变量与 trigger 失配）
- 循环依赖问题（A 模块依赖 B，只重载 A 可能破坏引用）

**实现路径**不简单：
1. Lua 端需要模块化重构（`require` 模式而非全局模式）
2. 内核需要跟踪每个 trigger/alias 来自哪个模块
3. 卸载旧模块时需要清理所有注册的 trigger/alias/timer

**建议**：先保证 `/load reload` 足够稳定（已有），精细化重载等脚本库本身模块化到一定程度再考虑。

**前置依赖**：脚本库模块化改造。

## 7. 群控 / 跨角色通信（Inter-Session Communication）

| 收益 | 风险 | 难度 | 急迫性 |
|:----:|:----:|:----:|:------:|
| ★★★ | ★★ | ★★★ | ★ |

**评估**：多个角色协作的场景很有想象力，但需求因人而异。不是所有用户都需要多角色配合。

**收益**：
- 实现多角色协同（支援、交易、组队）
- 一个角色的 Lua 脚本可以控制另一个角色
- 拓展 bot 玩法的上限

**风险**：中等。跨 session 通信如果设计不当可能导致：
- 循环触发（A 通知 B，B 通知 A，无限循环）
- 死锁（A 等 B，B 等 A）
- 调试困难（分布式问题）

**实现路径**：
1. Lua API: `session_send(target, cmd)` — 向另一 session 发送命令
2. Lua API: `session_broadcast(cmd)` — 向所有 session 广播
3. Lua 回调: `on_session_message(from_session_id, msg)`
4. 内核层通过 `mpsc` channel 转发消息

**前置依赖**：远程 API + 守护进程（群控的价值在 headless 模式下才凸显）。

## 8. Speedwalk 路径系统内置

| 收益 | 风险 | 难度 | 急迫性 |
|:----:|:----:|:----:|:------:|
| ★★ | ★ | ★★ | ★ |

**评估**：很多 MUD 脚本都需要走路，但通常每个脚本各自实现一套路径逻辑。内置统一解析器可以减少重复工作。但这不是一个"必须由客户端解决"的问题——Lua 脚本完全可以自己实现。

**收益**：
- 统一的路径格式和方向映射
- 减少脚本中的重复代码
- 可以和地图模块集成

**风险**：低。独立模块，不影响现有功能。

**实现路径**：
1. 定义路径文件格式（纯 Lua 或 TOML）
2. 内置方向缩写映射（n/north, s/south, e/east, w/west 等）
3. 实现 `Speedwalk("3n;2e;sw")` 解析器（其实就是个循环 + 间隔延迟）

**前置依赖**：无。

## 9. 插件系统 / 脚本模块化

| 收益 | 风险 | 难度 | 急迫性 |
|:----:|:----:|:----:|:------:|
| ★★★ | ★★★ | ★★★★★ | ★ |

**评估**：参考 MUSHclient 的插件机制，可以让多个独立作者开发的脚本共存互不冲突。这是一个"锦上添花"的功能——对作者社区很有吸引力，但对单人用户不是刚需。

**收益**：
- 脚本可复用、可分享
- 插件的 trigger/alias 有命名空间隔离，互不干扰
- 参考 MUSHclient 的 `AddPlugin` API，降低迁移成本

**风险**：高。
- 插件命名空间的设计会影响现有脚本架构
- 与当前"一个 session 一个脚本"的模式冲突
- 调试复杂性大幅上升

**实现路径**（需要大量设计）：
1. 定义插件清单格式（name, version, author, dependencies）
2. 实现命名空间隔离（插件 trigger 自动前缀或沙箱）
3. 实现插件间通信
4. 实现插件生命周期管理（enable/disable/reload）

**建议**：在当前版本不要碰，等脚本生态发展起来后再考虑。

**前置依赖**：远程 API、热更新、跨 session 通信等底层能力先就位。

## 10. Lua 执行沙箱（安全限制）

| 收益 | 风险 | 难度 | 急迫性 |
|:----:|:----:|:----:|:------:|
| ★★ | ★★★★ | ★★★★ | ★★ |

**评估**：对于"跑自己的脚本"场景，沙箱不是必须的——你信任自己的代码。但如果将来支持插件系统或多人共用客户端，沙箱就是安全底线。

**收益**：
- 防止脚本失控（死循环、内存泄漏、flood 服务器）
- 是插件系统安全性的前提
- 提升客户端稳定性

**风险**：高。mlua 的沙箱（`Safety` 模式、`hook` 回调）经过多年验证相对成熟，但：
- 超时中断（hook）可能打断 Lua GC，导致不可预测的状态
- 频率限制可能误伤合法操作
- 配置不当会让正常脚本跑不起来

**实现路径**：
1. 使用 mlua 的 `hook` 设置指令数限制（如每 tick 10 万条指令）
2. 超时时触发 `error()`，由 Lua 侧 catch 或客户端打印错误并继续
3. 命令发送队列增加速率限制（如最多 3 条/秒）
4. 所有限制做成可配置的（`GeneralConfig` 或 Lua `SetOption`）

**前置依赖**：无（但要谨慎上线，建议默认关闭）。

## 综合优先级矩阵

```
高收益 │  守护进程    远程 API
       │  健康监控    告警通知
       │  ★★★★★      ★★★★★
       │
       │  凭据管理    跨 session
       │  ★★★        ★★★
       │
       │  Speedwalk   热更新
       │  ★★          ★★★
       │              插件系统
       │              ★★★
低收益 │              沙箱
       │              ★★
       └───────────────────────────
        低风险 →          高风险
```

### 推荐迭代顺序

```
Phase 1 — 基础能力（本阶段收益最大）
  ├── 守护进程模式
  └── 远程控制 API（HTTP）

Phase 2 — 无人值守
  ├── 告警通知
  ├── 连接健康监控
  └── 凭据安全管理

Phase 3 — 进阶功能
  ├── 跨角色通信
  ├── Speedwalk 内置
  └── 脚本热更新（精细化重载）

Phase 4 — 生态建设
  ├── 插件系统
  └── Lua 沙箱安全限制
```
